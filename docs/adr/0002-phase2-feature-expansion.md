# 0002: Phase 2 功能扩展架构

## 状态

提议中

## 背景

通过对标原 Java 版 Moira (At Home Projects) 的功能，我们的 Tauri + Rust + React 版本在七政四馀排盘基础上缺少大量关键功能，包括四柱八字、大运、十神、长生十二宫等中国星命学核心内容。

## 决策

### 1. 历法数据管理

**问题**: 农历/节气相关计算需要大量历法数据（节气日期表、农历月大小、闰月等）。

**决策**: 采用 NASA 的农历计算方案，在 Rust 中实现基于天文参数的节气计算，而非硬编码历法数据表。

- **节气**: 基于太阳黄经到达特定度数（0°立春，15°惊蛰...）的计算。太阳位置已有 JPL DE440 数据。
- **农历**: 基于阴历月相周期 + 节气区间确定。
- **合理性**: 本项目已有 JPL 星历引擎，节气计算可直接复用现有设施。

### 2. 模块扩展策略

**问题**: 新增功能（八字、择日、西洋占星）本质上是独立子系统，不应耦合在同一个 lib.rs 中。

**决策**: 按功能域分拆为独立 .rs 文件，以模块方式组织，不新建 crate。

- `bazi.rs` — 四柱八字、大运、十神、藏干、长生十二宫、胎元
- `angles.rs` — 上升、天顶、福点
- `selection.rs` — 天星择日（后续）
- `western.rs` — 西洋占星（后续）

**替代考虑**:
- 新建独立 crate — 当前模块间数据共享频繁，独立 crate 增加接口成本
- 全部放进 lib.rs — 已接近 700 行，继续膨胀不可维护

### 3. 农历计算（P0 简化方案）

**问题**: 精确的农历转换需要完整的太阳黄经 → 节气判定，但第一期 P0 只需日柱/时柱的干支计算（与农历无关）。

**决策**: P0 的干支计算依赖公历日期即可完成：
- 年柱：以 1 月 1 日为界（简化），后续用立春修正
- 月柱：以公历月 1 日为界（简化），后续用节气修正
- 日柱：儒略日 → 干支（已有标准公式）
- 时柱：出生时辰 → 时干支（固定映射）

节气精确分割推迟到农历转换功能（P2）时实现。

### 4. 前端数据展示

**问题**: 新增数据量大（八字+大运+十神+长生），不应挤占星盘图空间。

**决策**:
- 星盘图保持不变（显示星曜位置 + 二十八宿 + 地支 + 宫位）
- 新增数据以**侧边抽屉**或**下方面板**形式展示
- 八字四柱显示在搜索栏下方或左侧表单区域内
- 大运/十神以表格展示
- 长生十二宫以环形图展示（可选）

### 5. 接口兼容性

**问题**: 新增字段会改变 `AstrologyData` 的结构，破坏现有前端。

**决策**: 
- 所有新增字段为 `Option<T>`，保持向后兼容
- 前端按需读取，不破坏现有渲染逻辑
- 现有 CLI main.rs 和 Tauri lib.rs 的 command 接口不变

## 后果

### 正面
- P0 可在不引入外部数据依赖的情况下完成
- 模块化设计便于后续维护和独立测试
- 向后兼容设计不会破坏现有功能

### 负面
- P0 的节气简化方案会引入精度误差（年柱若用 1/1 而非立春分界，约 5% 的日期年柱错误）
  → 计划在 P2 农历转换时修正
- 月柱顺逆排运当前默认为顺排，阴阳男女判断推迟到 P1
